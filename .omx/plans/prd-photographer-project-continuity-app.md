## Product Statement

这是一个面向长期摄影项目创作的结构化工具，帮助摄影师把数月到一年以上周期中的零散灵感、照片、元素和叙事关系持续组织成一个正在成形的摄影作品。

它不是照片管理工具，也不是通用笔记工具。它的核心价值是让摄影项目在长期创作中保持主题性、连贯性和可追踪的演化过程。

## Problem

长期摄影项目通常不是一次性完成，而是在很长时间里逐步补足、修正和生长出来的。摄影师在外拍中会不断被新刺激吸引，产生偏离原始主题的风险；回头整理时，又常常发现照片与原计划缺乏稳定映射，导致：

- 项目主题逐渐漂移
- 原先设想的重要元素没有被持续追踪
- 结构与内容之间的关系越来越模糊
- 零散想法难以回收成一个完整作品
- 外拍现场没有时间导入正式照片时，关键现场线索容易丢失
- 使用 Word / Markdown 一段时间后，资料容易变成大段文本堆，越来越难维护

## Target User

首要用户：

- 正在创作长期摄影项目的摄影师
- 创作对象可以是摄影画册、系列作品、章节式项目或具有明确主题的长期拍摄计划
- 用户强调主观创作判断，不希望工具替自己决定作品方向

暂不覆盖：

- 以修图、交付、选片、发布为主的摄影工作流
- 团队协作型工作室管理场景
- 客户项目管理场景

## Product Principles

1. 结构优先于堆积  
   工具首先要维持项目结构的清晰，而不是充当内容仓库。

2. 辅助判断，不替代判断  
   摄影师始终拥有创作决定权，工具不能主动判断偏题与否、不能替用户做作品判断。

3. 天生结构化，但不能更重  
   必须比 Word / Markdown 更结构化，但使用成本不能更高。

4. 双阶段工作流  
   外拍阶段只做低摩擦采集；回家后再做结构化整理与消项。

5. 轻交互，强骨架  
   交互必须轻，模型必须稳。第一版不做重型关系系统和大而全摄影平台。

## Core Outcome

用户在项目推进过程中，能够持续看到并维护：

- 项目的主题
- 项目的结构
- 项目的关键元素
- 外拍时形成的信息流节点
- 照片与结构/元素之间的映射
- 对比、呼应、重复、转折等关系属性
- 创作过程中的阶段性想法
- 项目如何随时间逐步成形

项目完成时，用户看到的不只是照片集合，而是：

- 一个具有章节感或组块感的作品结构
- 被逐步完成的元素系统
- 一条包含外拍记录与正式照片的现场信息流
- 具有呼应和对照关系的视觉组织
- 一条可回看的创作时间线

## Non-goals

MVP 明确不做：

- 照片编辑 / 修图
- 云盘备份
- 客户管理
- 作品发布
- 通用笔记系统
- 拍摄参数管理
- AI 自动选片
- AI 判断偏题
- AI 创作建议
- 社交协作

补充说明：

- 允许存在最基本的照片导入和必要编辑能力，但仅服务于项目构思闭环
- 不把产品做成 Lightroom、Notion 或文件管理器的替代品

## Conceptual Model

### 1. Project

摄影项目本体。

包含：

- 项目标题
- 主题陈述
- 项目描述
- 起止时间
- 当前阶段

说明：
- 主题不单独抽成复杂对象，先作为 `Project` 的核心字段存在

### 2. StructureNode

作品结构节点，代表摄影集里的章节、段落、组块或零散结构单元。

特点：

- 可以有顺序
- 也可以不要求线性顺序
- 负责表达作品的骨架

类比：
- 相当于散文的章节

### 3. Element

项目中需要被建立、反复出现或逐步补足的视觉/叙事材料。

特点：

- 一个元素可以由一张或多张照片共同完成
- 元素不是章节
- 元素是构成结构内容的一部分

类比：
- 相当于散文中的优美语句、过渡句、标点符号

### 4. Photo

被导入进项目的照片。

特点：

- 可与结构节点关联
- 可与元素关联
- 可带有文字说明
- 可带有关系属性

补充说明：

- 照片可以来自正式导入
- 也可以来自外拍时的快速拍摄记录

### 5. CaptureEntry

外拍信息流节点，用来承接现场快速记录。

特点：

- 以时间顺序存在
- 自动记录地点
- 可只包含文字
- 可包含一张快速拍摄照片
- 可包含一张临时从相册选入的照片
- 时间相近的记录会自动合并到同一节点

设计目的：

- 解决“现场先记一笔，回家再补正式照片”的业务问题

### 6. Relation

第一版不作为独立重型对象，而作为挂接时附加的关系属性存在。

示例：

- 呼应
- 对比
- 重复
- 转折

设计理由：

- 保留必要的叙事表达能力
- 避免把第一版做成复杂关系图谱系统

## Object Boundaries

### StructureNode 与 Element 的边界

二者严格分开，不混用。

- `StructureNode` 回答：这本摄影集由哪些部分组成？
- `Element` 回答：这些部分由哪些视觉/叙事材料构成？

原因：

- 混用会迅速让系统失去结构稳定性
- 严格分开更容易长期维护，也更符合“散文结构 vs 句子/标点”的创作心智

### Relation 的边界

第一版只做轻量关系属性，不做独立关系系统。

原因：

- 能表达必要的叙事连贯性
- 又不会把录入和维护成本推高

## Primary Workflow

### Phase A: 外拍采集

目标：

- 低摩擦记录现场素材、想法与环境上下文

允许动作：

- 写一段文字
- 直接拉起应用内相机拍一张
- 从相册临时选一张
- 以上内容进入按时间排序的信息流
- 自动记录地点

业务规则：

- 10 分钟内相近记录自动合并为同一个信息流节点
- 同一节点中可以逐步累加文字和临时照片
- 现场不要求立即导入正式相机照片

不要求：

- 现场完成结构化挂接
- 现场消项
- 现场梳理完整关系

### Phase B: 回家整理

目标：

- 把采集到的内容组织回项目叙事

核心动作：

- 导入正式照片
- 系统按时间自动关联到最近的信息流节点
- 用户可手动把照片重新关联到其他时间节点
- 把照片挂接到结构节点
- 把照片挂接到元素
- 在挂接时附加关系属性
- 对已满足的元素进行消项
- 补充创作阶段想法和关联思路
- 调整项目结构、元素和照片的对应关系

## MVP Functional Scope

### Must Have

1. 创建项目并定义主题
2. 创建结构节点
3. 创建元素
4. 外拍时创建信息流节点
5. 外拍时记录文字
6. 外拍时快速拍一张或临时选一张放入信息流
7. 自动记录地点
8. 10 分钟内的相近记录自动合并为同一信息流节点
9. 导入正式照片到项目
10. 系统按时间自动把导入照片关联到最近的信息流节点
11. 允许用户手动重新关联照片到其他信息流节点
12. 将照片关联到结构节点
13. 将照片关联到元素
14. 在挂接时标记关系属性
15. 对元素进行消项
16. 查看项目总览
17. 查看创作时间线与外拍信息流

### Should Have

1. 允许一个元素被多张照片共同满足
2. 允许同一张照片同时关联结构节点和多个元素
3. 允许用户在整理阶段补充新的元素
4. 允许用户补充新的关联思路
5. 在总览视图中看到“哪些元素已完成 / 未完成”
6. 允许用户修正自动匹配到的信息流节点

### Not in MVP

1. 自动推荐缺失元素
2. 自动判断偏题
3. 多人协作
4. 云同步 / 备份体系
5. 发布 / 输出成册流程
6. 高级照片管理与筛选体系

## Information Architecture

第一版建议收敛为 4 个核心工作面：

### 1. 项目主页

负责展示：

- 项目主题
- 当前进度
- 最近整理记录
- 结构与元素的摘要状态

### 2. 结构 / 元素总览板

负责展示：

- 结构骨架
- 每个结构节点下的元素状态
- 已完成 / 未完成元素
- 照片与元素之间的完成度关系

### 3. 照片导入与挂接面板

负责完成：

- 照片导入
- 写简短说明
- 关联结构节点
- 关联元素
- 标记关系属性
- 消项

要求：

- 尽量在一个工作面内完成核心动作
- 避免来回跳多个页面

### 4. 创作时间线

负责展示：

- 项目从开始到当前的演化过程
- 哪些元素在何时被建立
- 哪些照片在何时进入项目
- 哪些创作思路在何时出现和变化

## User Experience Constraints

第一版必须满足以下 UX 约束：

- 不能要求用户先搭复杂结构后才能开始记录
- 不能要求用户在现场完成高认知负担整理
- 不能让用户为了挂接一张照片频繁切换多个页面
- 不能把关系系统做成复杂图数据库交互
- 不能让用户感觉比 Word / Markdown 更麻烦

## Success Metrics

这不是增长型 MVP，因此成功信号以创作质量和持续使用能力为主。

首批用户成功信号：

- 用户能在 3 个月内持续维护项目，不退回 Word / Markdown
- 用户能清楚看到项目结构和元素完成状态
- 用户能回看项目是如何演化的
- 用户认为项目整体更连贯、更直观
- 用户认为这个工具有助于保持主题专注，而不是让材料越来越乱

## Risks

1. 对象模型过抽象  
   用户不知道结构和元素该怎么建，导致上手卡住。

2. 挂接流程过重  
   用户拍完后不愿意整理，闭环断掉。

3. 总览视图不够直观  
   产品差异化丧失，退化为结构化笔记工具。

4. 第一版范围失控  
   一旦引入 AI、资产管理、协作等模块，产品会变重。

## ADR

### Decision

将产品规划为“轻交互、强结构”的摄影项目创作工作台，而不是照片管理或重型知识系统。

### Drivers

- 长期摄影项目容易主题漂移和结构失序
- Word / Markdown 缺乏原生结构化表达
- 用户不能接受高复杂度工具

### Alternatives Considered

- 做成照片管理工具  
  被拒绝，因为会偏离项目构思核心。

- 做成通用知识管理工具  
  被拒绝，因为结构会依赖用户手工维护，最终仍会混乱。

- 把 Relation 做成独立关系系统  
  被拒绝，因为第一版复杂度过高。

### Why Chosen

这个方向最符合用户心智：像构思一本摄影集，而不是管理一个图库。

### Consequences

- 第一版会更聚焦
- 建模需要非常清晰
- 页面数量和交互流程必须被严格控制

### Follow-ups

- 在下一阶段明确低保真页面骨架
- 验证结构节点与元素的创建方式是否足够自然
- 验证挂接 + 消项是否能在单工作面内顺畅完成
