## IA Goal

本信息架构的目标是在纯手机端前提下，仍然支撑以下闭环：

1. 建立项目主题与结构
2. 管理元素
3. 建立外拍信息流节点
4. 导入照片
5. 将照片挂接到结构和元素
6. 对元素进行消项
7. 回看项目整体与创作时间线

它必须同时满足：

- 小屏可用
- 结构清晰
- 操作负担低
- 不比 Word / Markdown 更复杂

## IA Principles

1. 以项目为中心，而不是以照片库为中心  
   用户进入的是一个摄影项目，而不是素材仓库。

2. 以任务流为中心，而不是以复杂导航为中心  
   手机端不适合层层信息面板，更适合围绕任务组织页面。

3. 以分步整理替代桌面式并排工作面  
   小屏不能照搬桌面三栏结构，整理动作应拆成清晰的步骤流。

4. 持续强化“结构”和“元素”的区分  
   导航、文案和页面结构必须避免两者混淆。

5. 时间线是回看视角，不是主编辑入口  
   时间线应帮助理解项目如何形成，而不是承担核心编辑职责。

## Top-Level Navigation

第一版建议使用 4 个一级入口，适合底部导航或顶层 Tab：

1. `项目`
2. `结构`
3. `整理`
4. `历程`

说明：

- `项目` 对应项目主页
- `结构` 对应结构与元素工作区
- `整理` 对应照片导入与挂接主流程
- `历程` 对应创作时间线

不建议第一版加入一级导航：

- 相册库
- 文件中心
- Web 入口
- AI 助手
- 发布中心
- 协作中心

## Page Map

### 1. 项目主页

目标：
- 让用户一进入项目，就知道这个项目当前是什么、进展到哪里、下一步该去哪里

核心内容：
- 项目标题
- 主题陈述
- 项目简介
- 当前阶段
- 结构进度摘要
- 元素完成状态摘要
- 待整理照片数量
- 最近创作想法

用户在这里最常做的动作：
- 回看主题
- 进入结构页
- 进入整理页
- 进入时间线

移动端要求：
- 内容以卡片和模块化区块呈现
- 首屏优先露出主题、进度和“继续整理”入口

### 2. 结构页

目标：
- 呈现项目骨架和内容材料之间的关系

核心内容：
- 结构节点列表
- 元素列表
- 已完成 / 未完成元素筛选
- 结构节点详情
- 元素详情

重要说明：
- 结构和元素不在一个混合列表里展示
- 推荐采用“分段切换”或“二级 Tab”来切换结构与元素，而不是同屏并排

用户在这里最常做的动作：
- 新建结构节点
- 调整结构顺序
- 新建元素
- 查看某个结构节点下有哪些元素
- 查看某个元素由哪些照片支撑

### 3. 整理页

目标：
- 承担 MVP 的核心生产流程

核心内容：
- 外拍信息流节点
- 待整理照片流
- 当前选中照片
- 结构挂接
- 元素挂接
- 关系属性
- 元素消项
- 创作想法记录

这是第一版最关键的工作面。

移动端要求：
- 不做桌面式多栏
- 采用“当前照片 + 分步编辑区”的任务流
- 允许通过底部弹层、分段卡片或步骤面板完成挂接
- 必须能看到“信息流节点”和“后续导入照片”的连接关系

### 4. 创作时间线

目标：
- 回看项目如何一步步成形

核心内容：
- 照片导入事件
- 元素完成事件
- 结构调整事件
- 创作想法记录事件
- 阶段性变化节点

用户在这里最常做的动作：
- 回看某个时间段发生了什么
- 理解项目如何演化
- 打开某次事件对应的照片或想法

## Primary User Flows

### Flow A: 新建项目

路径：
1. 创建项目
2. 填写主题陈述
3. 输入项目简介
4. 进入结构页
5. 创建初始结构节点
6. 创建初始元素

结束状态：
- 项目进入可持续拍摄和整理状态

### Flow B: 外拍信息流记录

路径：
1. 打开整理页
2. 新建一条信息流记录
3. 写文字，或快速拍一张，或从相册临时选一张
4. 系统自动记录时间与地点
5. 10 分钟内的相近记录自动并入同一信息流节点

结束状态：
- 项目中形成一条可回看的外拍信息流节点

说明：
- 此时不强迫挂接结构与元素
- 此时也不要求导入正式照片

### Flow C: 回家整理

路径：
1. 打开整理页
2. 导入正式照片
3. 系统按时间自动关联到最近的信息流节点
4. 用户可手动修正归属节点
5. 选择一张待整理照片
6. 关联到结构节点
7. 关联到元素
8. 设置关系属性
9. 视情况对元素消项
10. 写下当前创作想法
11. 保存并进入下一张

结束状态：
- 照片被纳入项目叙事
- 元素完成状态被更新

### Flow D: 回看项目

路径：
1. 打开时间线
2. 查看某次导入或整理事件
3. 打开对应照片、元素或想法
4. 如需要，回到整理页继续处理

结束状态：
- 用户理解项目是如何逐步形成的

## Navigation Rules

1. `项目主页` 是项目上下文的入口页。
2. `整理页` 是第一版核心工作页，应能从项目主页快速进入。
3. `结构页` 负责看清结构和元素，不承担完整照片整理流。
4. `创作时间线` 只承担回看职责，不承担主要编辑职责。
5. 第一版不做复杂层级导航，不做深层信息树。

## Suggested Cross-Links

- 项目主页 -> 继续整理 -> 整理页
- 项目主页 -> 查看结构 -> 结构页
- 项目主页 -> 查看历程 -> 创作时间线
- 结构页中的元素详情 -> 查看支撑照片 -> 整理页
- 创作时间线中的事件 -> 打开对应照片 / 元素 / 想法

## IA Constraints

必须满足：

- 结构与元素是不同工作对象，界面不能混成一种卡片体系
- 整理页是第一版核心页面，不能被削弱成附件页
- 用户必须能在较少步骤内完成挂接和消项
- 用户必须能快速建立外拍信息流，而不依赖正式照片导入
- 时间线必须服务“回看演化”，而不是变成杂乱日志

必须避免：

- 照搬桌面三栏结构到手机端
- 以照片库为中心设计整套产品
- 把结构、元素、关系做成三个独立复杂系统
- 为了完整性加入太多页面和层级

## Open IA Decisions for Next Iteration

1. 结构页更适合用二级 Tab 还是分段切换来区分“结构 / 元素”？
2. 整理页更适合单张顺序整理，还是支持小批量整理？
3. 项目主页是否需要突出“待整理数量”作为第一优先信息？
4. 时间线要不要按“导入 / 整理 / 完成元素”分组展示？
