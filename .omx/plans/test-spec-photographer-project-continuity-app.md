## Purpose

本测试规格不是代码级测试清单，而是产品规划阶段的验收与验证标准，用来判断这套方案是否真正满足摄影项目创作场景，而不是退化成另一个通用笔记或照片管理工具。

## Validation Strategy

按四类验证：

1. 对象模型是否清晰
2. 核心闭环是否能完成
3. 交互复杂度是否可接受
4. 产品差异化是否成立

## Acceptance Criteria

### A. 对象模型

1. 用户可以创建一个项目，并为其填写主题。
2. 用户可以为项目创建结构节点。
3. 用户可以为项目创建元素。
4. 用户能够理解结构节点与元素的区别，而不需要依赖额外培训。
5. 结构节点与元素在界面和数据模型中是严格分开的。
6. 关系作为轻量属性存在，不要求用户维护独立关系对象。

### B. 外拍采集闭环

1. 用户在外拍时可以快速创建信息流节点。
2. 用户在外拍时可以只写文字，不必先导入正式照片。
3. 用户在外拍时可以直接用应用拍一张，或从相册临时选一张加入信息流。
4. 系统会自动记录该信息流节点的时间与地点。
5. 10 分钟内相近记录会自动合并为同一信息流节点。
6. 用户在外拍时不需要完成结构化挂接，也不会被强迫消项。
7. 现场采集流程应在低认知负担下完成。

### C. 回家整理闭环

1. 用户可以在整理阶段导入正式照片。
2. 系统可以按时间自动将导入照片关联到最近的信息流节点。
3. 用户可以手动把照片重新关联到其他信息流节点。
4. 用户可以在整理阶段将照片关联到结构节点。
5. 用户可以在整理阶段将照片关联到一个或多个元素。
6. 用户可以在整理阶段为挂接增加关系属性，例如呼应、对比、重复、转折。
7. 用户可以把已满足的元素标记为完成。
8. 用户可以在整理阶段补充新的元素或创作思路。
9. 完成一次整理时，不需要在过多页面中反复切换。

### D. 总览与时间线

1. 用户可以看到项目结构的整体视图。
2. 用户可以看到元素的完成状态。
3. 用户可以看到信息流节点与正式照片之间的对应关系。
4. 用户可以看到照片如何支撑结构与元素。
5. 用户可以看到项目的创作时间线。
6. 用户能够回看某个阶段的创作思路和项目演化。

### E. 差异化成立

1. 用户认为该工具比 Word / Markdown 更容易保持结构清晰。
2. 用户认为该工具不会比 Word / Markdown 更复杂。
3. 用户认为该工具不会破坏自己的创作主观判断。
4. 用户认为该工具帮助项目保持连贯，而不是单纯堆积素材。

## Negative Checks

以下任何一项成立，都说明方案偏离目标：

1. 用户把它当成照片仓库使用，而不是创作结构工具。
2. 用户不知道结构节点和元素的区别。
3. 用户在现场不能快速留下一个信息流节点。
4. 用户需要复杂步骤才能完成一次照片挂接。
5. 用户需要先建大量结构才能开始记录。
6. 用户觉得它比 Word / Markdown 更麻烦。
7. 用户误以为系统会替自己判断作品方向。

## Manual Validation Scenarios

### Scenario 1: 新建项目

目标：
- 验证用户能否从主题进入结构和元素建模

步骤：
1. 新建一个长期摄影项目
2. 输入主题陈述
3. 建立 3 到 5 个结构节点
4. 建立 5 到 10 个元素

通过标准：
- 用户知道自己在建“章节骨架”还是“内容材料”
- 用户没有把结构与元素混在一起

### Scenario 2: 外拍记录

目标：
- 验证现场录入是否足够轻

步骤：
1. 在外拍中快速记录两到三次现场信息
2. 一次只写文字
3. 一次只拍一张临时照片
4. 一次从相册选一张并加一段文字
5. 检查 10 分钟内的记录是否被自动合并

通过标准：
- 用户能快速完成
- 用户不觉得被流程拖住
- 用户能看到时间和地点被自动保留

### Scenario 3: 回家整理

目标：
- 验证挂接与消项闭环

步骤：
1. 导入一次外拍后的正式照片
2. 检查系统是否自动把照片关联到最近的信息流节点
3. 手动修正其中一两张照片的信息流归属
4. 把若干照片挂接到结构节点
5. 把若干照片挂接到元素
6. 为部分挂接补充关系属性
7. 对已满足的元素执行消项

通过标准：
- 用户能自然完成挂接
- 用户能理解“信息流节点 -> 正式照片”的连接逻辑
- 用户能理解哪些元素已被满足
- 用户不需要跨太多页面切换

### Scenario 4: 项目回看

目标：
- 验证产品是否真的帮助长期项目保持连贯

步骤：
1. 查看项目总览
2. 查看元素完成情况
3. 查看创作时间线
4. 回看某个时间点的创作思路

通过标准：
- 用户能迅速理解项目当前状态
- 用户能感知项目如何逐步成形

## Open Questions for Next Stage

1. 结构节点的创建方式更适合树状、卡片式还是时间顺序式？
2. 元素的状态是简单“未完成/完成”，还是需要更多阶段状态？
3. 关系属性最少保留哪几种，才能既够用又不复杂？
4. 照片挂接与消项是否应该在同一个工作面完成？
5. 时间线是以照片进入项目为主，还是以创作事件为主？

## Pre-mortem

### Failure 1

产品建模太抽象，用户初次使用就不知道怎么下手。

缓解：
- 提供非常轻量的默认模板
- 在界面中持续强化“结构 vs 元素”的区分

### Failure 2

整理成本过高，用户拍完后懒得做挂接。

缓解：
- 保证挂接、消项、写想法可在同一工作面完成
- 控制操作步骤，避免复杂配置

### Failure 3

总览视图不具备明显优势，用户回退到 Word / Markdown。

缓解：
- 总览板必须能直观看到结构、元素和完成状态
- 时间线必须清楚呈现项目演化，而不是简单日志

## Verification Recommendation

下一阶段不应先写代码，应先做：

1. 低保真信息架构稿
2. 核心页面骨架
3. 一轮以摄影师创作心智为中心的交互走查

只有当以下问题被验证后，才适合进入实现规划：

- 结构和元素的区分是否自然
- 挂接与消项是否顺手
- 时间线是否真正有价值
- 总览视图是否足够直观
